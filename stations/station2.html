CSS Custom Properties to the Rescue

Then Prof. Teeters showed me CSS custom properties. Instead of #3366cc everywhere, I now had --primary. Want to change it? Update ONE line! The syntax is super easy - if you can write CSS, you can use custom properties.

Define in :root

Makes them available everywhere:

:root {
  --primary: hsl(210, 50%, 50%);
}

Use with var()

Anywhere you'd use a color:

.button {
  background: var(--primary);
}

That's it! But here's the real magic - watch how this transforms my hex soup disaster.

BEFORE - Hex Soup Disaster (This is literally from my old code):
.header { background: #3366cc; color: white; }
.nav { background: #3367cc; color: #ffffff; }
.button { background: #3366CC; color: #FFF; }
.card { border: 1px solid #3366cd; }
.hero { background: #3366cb; }
.footer { background: #3366cc; color: white; }
.link { color: #3366CC; }
.link:hover { color: #2255bb; }
.tag { background: #3367cd; }
.badge { background: #3366cc; }
/* ... and 37 more variations ... */

Count the different blues! I found 8 slightly different values. Some were probably the same color with case differences. I couldn't tell you which was correct!

AFTER - Custom Properties (What I Should Have Done):
:root {
  --primary: hsl(210, 50%, 50%);
  --primary-dark: hsl(210, 50%, 40%);
  --primary-light: hsl(210, 50%, 60%);
  --on-primary: hsl(0, 0%, 100%);
}

.header { background: var(--primary); color: var(--on-primary); }
.nav { background: var(--primary); color: var(--on-primary); }
.button { background: var(--primary); color: var(--on-primary); }
.card { border: 1px solid var(--primary); }
.hero { background: var(--primary); }
.footer { background: var(--primary); color: var(--on-primary); }
.link { color: var(--primary); }
.link:hover { color: var(--primary-dark); }
.tag { background: var(--primary-light); }
.badge { background: var(--primary); }

Client wants a different blue? Change the hue and saturation values in all three variables. Better than hunting through 47 hex codes, but there's still a smarter way...

üü† Advanced Technique - HSL Component Variables

But Prof. Teeters showed me an even smarter way that professional developers use to max out HSL's capabilities:

:root {
  /* Base hue, saturation, and lightness */
  --primary-h: 210;
  --primary-s: 50%;
  --primary-l: 50%;

  /* Adjustable steps for lightness */
  --primary-l-dark-step: 10%;
  --primary-l-light-step: 10%;

  /* Derived colors */
  --primary: hsl(var(--primary-h), var(--primary-s), var(--primary-l));
  --primary-dark: hsl(var(--primary-h), var(--primary-s), calc(var(--primary-l) - var(--primary-l-dark-step)));
  --primary-light: hsl(var(--primary-h), var(--primary-s), calc(var(--primary-l) + var(--primary-l-light-step)));

  --on-primary: hsl(0, 0%, 100%);
}

Yeah, those lines look messy and are hard to read but we have moved into some serious theming now.

Client wants green instead of blue? Change ONE line (the --primary-h definition). Done! Every instance updates automatically!

Client wants darker green for button hovers? Change the step value. One line. One and done!

This is next-level theming, but you don't need it when you're starting out. The basic version works great!

Creating Systematic Scales

The real power of custom properties is systematic scales. Instead of random grays everywhere, create a scale:

Organized Gray Scale:
:root {
  --gray-100: hsl(0, 0%, 95%);   /* Lightest */
  --gray-200: hsl(0, 0%, 85%);
  --gray-300: hsl(0, 0%, 75%);
  --gray-400: hsl(0, 0%, 65%);
  --gray-500: hsl(0, 0%, 50%);   /* Middle */
  --gray-600: hsl(0, 0%, 40%);
  --gray-700: hsl(0, 0%, 30%);
  --gray-800: hsl(0, 0%, 20%);
  --gray-900: hsl(0, 0%, 10%);   /* Darkest */
}

Now you have consistent grays! Light background? --gray-100. Muted text? --gray-600. Dark text? --gray-900. So organized!

Building a Complete System

Prof. Teeters taught me to start with these core custom properties:

Complete Color System:
:root {
  /* Brand colors */
  --primary: hsl(210, 70%, 50%);
  --primary-hover: hsl(210, 70%, 40%);
  --primary-light: hsl(210, 70%, 90%);
  --primary-dark: hsl(210, 70%, 30%);

  --secondary: hsl(280, 50%, 50%);
  --secondary-hover: hsl(280, 50%, 40%);

  /* Neutral scale */
  --gray-100: hsl(0, 0%, 95%);
  --gray-500: hsl(0, 0%, 50%);
  --gray-900: hsl(0, 0%, 10%);

  /* Semantic colors */
  --success: hsl(120, 40%, 40%);
  --warning: hsl(45, 90%, 50%);
  --error: hsl(0, 65%, 50%);

  /* Text colors */
  --text-primary: var(--gray-900);
  --text-secondary: var(--gray-500);
  --text-on-primary: hsl(0, 0%, 100%);

  /* Background colors */
  --bg-page: hsl(0, 0%, 98%);
  --bg-card: hsl(0, 0%, 100%);
}

Notice how I reference other custom properties with var()! --text-primary uses --gray-900. That's allowed! If I change --gray-900, --text-primary updates too!

üî¨ Try This: Refactor Your Hex Chaos

Take any old CSS with scattered hex codes and convert it:

  1. Find all unique colors (ignore case, shorthand)
  2. Group similar colors (your different blues probably should be the same blue)
  3. Name them semantically (--primary, not --blue)
  4. Create scales where needed (gray-100 through gray-900)
  5. Replace all hex codes with var(--name)
  6. Test by changing ONE custom property and watching everything update!

Prof. Teeters had me do this with my worst project. 47 colors became 12 custom properties. TWELVE!

The Power of Semantic Naming

Custom properties were great, but I was still naming them badly! I had --blue, --dark-blue, --light-blue. Then the client said "change the blue to purple." Now my CSS had variables called --blue that were actually PURPLE! The variable was lying!

Prof. Teeters taught me semantic naming: describe the purpose, not the appearance.

Why Appearance Names Fail

My old naming was all appearance-based:

Appearance-Based Naming (The Problem):
:root {
  --blue: hsl(210, 50%, 50%);
  --dark-blue: hsl(210, 50%, 40%);
  --red: hsl(0, 65%, 50%);
}

.button { background: var(--blue); }
.link { color: var(--dark-blue); }
.error { color: var(--red); }

Then the client said "change the blue to purple." Now I have:

:root {
  --blue: hsl(270, 50%, 50%);  /* This is PURPLE! */
}

The variable is lying! Anyone reading my code is confused! Prof. Teeters said "names should describe purpose, not appearance."

Semantic Naming to the Rescue

The semantic version:

Semantic Naming (The Solution):
:root {
  --primary: hsl(210, 50%, 50%);
  --primary-dark: hsl(210, 50%, 40%);
  --error: hsl(0, 65%, 50%);
}

.button { background: var(--primary); }
.link { color: var(--primary-dark); }
.error { color: var(--error); }

Client wants purple? Change the hue in both --primary and --primary-dark to purple! The names still make sense because they're STILL the primary brand color. The code doesn't lie! (And yes, with the advanced HSL component technique from Section 1, you really would only change ONE line!)

Semantic Naming Patterns

Prof. Teeters taught me these naming patterns:

Brand Colors

--primary        /* Main brand color */
--secondary      /* Supporting brand color */
--accent         /* Pop of color for emphasis */

State Colors

--success        /* Successful actions */
--warning        /* Caution */
--error          /* Problems */
--info           /* Neutral information */

Text Colors

--text-primary   /* Main content text */
--text-secondary /* Muted text */
--text-disabled  /* Inactive text */
--text-on-primary /* Text on colored backgrounds */

Background Colors

--bg-page        /* Overall page background */
--bg-card        /* Card/panel backgrounds */
--bg-overlay     /* Modal overlays */

Border Colors

--border-subtle  /* Light dividers */
--border-strong  /* Emphasized borders */

The 60-30-10 Design Rule

Prof. Teeters taught me a secret formula that instantly made my designs look professional: the 60-30-10 rule. 60% dominant color, 30% secondary color, 10% accent color. Before this, I'd use equal amounts of all my colors and wonder why everything looked chaotic and busy!

What is 60-30-10?

60% Dominant

Your main color, usually a neutral like white, cream, or light gray. This is backgrounds, large areas. It creates calm and space!

30% Secondary

Your supporting color, creates contrast with the dominant. This is sidebars, navigation, secondary sections. It adds interest without overwhelming!

10% Accent

Your "pop" color, draws the eye to important things. This is buttons, links, calls-to-action. A little goes a LONG way!

Prof. Teeters said "think of a room: 60% walls, 30% furniture, 10% art and accessories." That visual helped me SO much!

How I Was Violating It

My old approach was chaos. "I have 5 colors, so I'll use each about 20%!"

Result? Visual noise! Everything screamed for attention! Nothing was important because EVERYTHING was emphasized!

Prof. Teeters showed me a site I'd designed: 20% blue header, 20% orange sidebar, 20% green content area, 20% purple footer, 20% red buttons. Her exact words: "It looks like you're trying to use every crayon in the box." Ouch! But accurate!

BAD - Equal Distribution (My Old Way):
:root {
  --color-1: hsl(210, 50%, 50%);  /* Blue */
  --color-2: hsl(30, 70%, 50%);   /* Orange */
  --color-3: hsl(150, 50%, 50%);  /* Green */
  --color-4: hsl(270, 50%, 50%);  /* Purple */
  --color-5: hsl(0, 70%, 50%);    /* Red */
}

.header { background: var(--color-1); }        /* 20% blue */
.sidebar { background: var(--color-2); }       /* 20% orange */
.main { background: var(--color-3); }          /* 20% green */
.footer { background: var(--color-4); }        /* 20% purple */
.button { background: var(--color-5); }        /* 20% red */
/* It's a rainbow explosion! */
GOOD - 60-30-10 Distribution (What I Learned):
:root {
  /* 60% - Dominant (neutral, calm) */
  --dominant: hsl(0, 0%, 97%);
  --dominant-alt: hsl(0, 0%, 100%);

  /* 30% - Secondary (your brand color) */
  --secondary: hsl(210, 50%, 50%);
  --secondary-dark: hsl(210, 50%, 40%);

  /* 10% - Accent (pop of contrast) */
  --accent: hsl(30, 80%, 55%);
  --accent-hover: hsl(30, 80%, 45%);
}

/* 60% usage */
body { background: var(--dominant); }
.card { background: var(--dominant-alt); }
.content-area { background: var(--dominant); }

/* 30% usage */
.header { background: var(--secondary); }
.sidebar { background: var(--secondary); }
.section-heading { color: var(--secondary-dark); }

/* 10% usage */
.button-primary { background: var(--accent); }
.button-primary:hover { background: var(--accent-hover); }
.link { color: var(--accent); }
.highlight { background: var(--accent); }

See the difference? The 60-30-10 version has hierarchy! The dominant color creates calm space. The secondary color defines structure. The accent draws your eye to important actions. Visual harmony!

Applying the Rule to Your Variables

When you create your color system, think about usage distribution:

  • 2-3 dominant/neutral custom properties (backgrounds, large areas = 60% of your site)
  • 2-3 secondary/brand custom properties (navigation, structure = 30% of your site)
  • 1-2 accent custom properties (buttons, links, CTAs = 10% of your site)

Don't create 20 custom properties if you only need 8! Prof. Teeters says "custom properties should make decisions easier, not harder."

üî¨ Try This: Audit Your Color Usage

Open your browser DevTools on any site and check color usage:

  1. What percentage is light/neutral backgrounds? (Should be ~60%)
  2. What percentage is your brand/structural color? (Should be ~30%)
  3. What percentage is attention-grabbing accent? (Should be ~10%)

Prof. Teeters had me do this on professional sites. They ALL followed 60-30-10! Then I checked my own sites. I was doing 20-20-20-20-20. No wonder they looked amateur!

üéì HyBit's Quick Reference

1

Start with 8 Custom Properties Maximum

Best for: Beginners who need simplicity and focus. Start with --primary, --secondary, --success, --error, --text, --background, --border, --accent. You can always add more later, but starting small prevents overwhelm!

2

Use the Numbering System for Scales

Best for: Grays and tints/shades. The 100-900 system (like Tailwind) is intuitive. 100 = lightest, 900 = darkest. Everyone understands it!

--gray-100: hsl(0, 0%, 95%);
--gray-500: hsl(0, 0%, 50%);
--gray-900: hsl(0, 0%, 10%);

Worst for: Brand colors where you only need one shade. Don't do --primary-500 if you're not making a scale!

3

Document Your System

Best for: Future you and your teammates! Add comments explaining your color system right in the CSS:

:root {
  /* === BRAND COLORS === */
  /* Primary: Main brand color, used for buttons and key UI */
  --primary: hsl(210, 70%, 50%);

  /* === NEUTRAL SCALE === */
  /* Gray scale: 100 (lightest) to 900 (darkest) */
  --gray-100: hsl(0, 0%, 95%);
  --gray-900: hsl(0, 0%, 10%);
}

You'll thank yourself in 6 months when you can't remember why you chose these colors!

4

Custom Properties Can Reference Each Other

Quick tip: Let custom properties reference each other using var() to keep things DRY!

:root {
  --gray-900: hsl(0, 0%, 10%);
  --text-primary: var(--gray-900);  /* References another variable! */
}

Change --gray-900 and --text-primary updates automatically! Prof. Teeters calls this "cascading custom properties."
